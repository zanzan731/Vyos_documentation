\section{Naloga in zasnova omrežja}
V tej dokumentaciji opišemo nalogo podjetja in predlagano omrežno zasnovo. Podjetje je start-up brez obstoječe IT infrastrukture; cilj je postaviti ločena omrežna segmenta za uporabnike in za strežnike, zagotoviti varen dostop do interneta in izpostaviti le izbrane javne storitve.

\subsection{Cilji naloge}
\begin{itemize}
  \item Postaviti omrežje z ločenimi segmenti za \textbf{uporabnike} in \textbf{strežnike} (strogo ločeno prometno okolje).
  \item Vsi strežniki so gostovani v DMZ (po zahtevah naloge) — javne in interne storitve bodo fizično v DMZ, vendar z omejitvami dostopa.
  \item Izpostaviti navzven le izbrane storitve: WireGuard, \texttt{wg-portal} (HTTPS) in REST frontend (HTTPS).
  \item Kritične administrativne in imenik storitve (AD, DNS, SNMP) dostopne samo znotraj omrežja in preko VPN (ni direktnega WAN dostopa).
  \item Ohranjati poseben \textbf{ipv6only} segment za eksperimentalne/izobraževalne potrebe z NPTv6 preslikavo za odhodni promet.
\end{itemize}

\subsection{Opredelitev segmentov}
\begin{itemize}
  \item \textbf{DMZ (eth1, 192.168.X.0/24)}: vsi strežniki (REST, wg-portal, WireGuard endpoint, AD/DNS, SNMP, ostali). Fizični lokaciji strežnikov sta v DMZ, vendar dostopi do nekaterih storitev omejeni z omrežnimi ACL.
  \item \textbf{INTERNAL (eth2, 10.X.0.0/24)}: uporabniške delovne postaje, administrativne postaje in monitoring hosti. Od tu je dovoljen nadzorovan dostop do AD/DNS/SNMP v DMZ.
  \item \textbf{IPV6ONLY (eth3, ULA + NPTv6)}: eksperimentalni IPv6-only segment; omogoča učenje in testiranje IPv6 funkcionalnosti. Izhodni promet je preko NPTv6 preslikave na javni IPv6.
\end{itemize}

\subsection{Izolacija segmentov}
Segmenti se izolirajo z naslednjimi ukrepi:
\begin{itemize}
  \item \textbf{Layer-3 ločitev (VyOS routing + ACL)}: vsak segment ima ločen subnet in VyOS izvaja routing ter aplikacijo politk (default-deny ter strog seznam izjem).
  \item \textbf{VLAN/PortGroup} na hipervizorju: DMZ, INTERNAL in IPV6ONLY so v ločenih PortGroup/VLANih, management port group pa ločen in omejen.
  \item \textbf{Host-firewall} (vsak strežnik in vsaka delovna postaja): default-deny inbound, dovoljenje samo za nujne storitve; admin dostop omejen na internal in VPN izvore.
  \item \textbf{Split DNS / conditional forward}: notranji klienti uporabljajo AD DNS za `startup11.local`, zunanji resolverji pa javne strežnike — prepreči izpostavitev notranjih zapisov.
  \item \textbf{Monitoring in logging}: monitoring host ima izključno dovoljenje za SNMP/agent promet; centralizirano beleženje (syslog) in netflow/sflow za analizo.
\end{itemize}

Split DNS je nastavljen tako, da se poizvedbe za \texttt{startup11.local} ne pošiljajo na javne resolverje, ampak na notranji AD DNS strežnik. To pomeni:
\begin{itemize}
  \item VyOS za domeno \texttt{startup11.local} uporablja conditional forward na AD DNS \texttt{192.168.11.201} oziroma \texttt{2001:1470:fffd:a9::201};
  \item AD DNS ostane avtoritativni vir za notranje zapise, predvsem za SRV zapise, ki so potrebni pri prijavi v domeno;
  \item vse ostale poizvedbe grejo prek upstream strežnikov iz prejšnje tabele;
  \item stari Linux DNS strežnik na \texttt{192.168.11.105} trenutno ni v uporabi; lokalne DNS zahteve zato gredo neposredno na notranji AD DNS server.
\end{itemize}

\section{NAT in preusmeritve vrat}

\paragraph{Namen}
Kratek, pregleden povzetek aktivnih NAT pravil: DNAT (port forwarding) za izbrane javne storitve, SNAT/masquerade za odhodni promet ter NAT66 (NPTv6) za IPv6-only odjemalce. Hairpin DNAT/SNAT omogoča, da notranji ali DMZ klienti dosežejo storitve preko javnega WAN naslova.

\subsection{Opombe in primeri}
\begin{itemize}
  \item DNAT in pripadajoča forward pravila so omejena na specifične porte in ciljne gostitelje — ni splošnega "port forward all" vedenja.
  \item Hairpin (DNAT+SNAT) obstaja samo za izbrane storitve (primarno HTTP/TLS porti), tako da notranji promet, ki cilja na WAN IP, pravilno doseže DMZ gostitelje.
  \item SNAT (masquerade) je uporabljen izključno za odhodni promet preko \texttt{eth0}; interni in DMZ segmenti ohranjajo ločeno notranje naslavljanje.
\end{itemize}

\section{Firewall}
\subsection{Kratek povzetek}
\begin{itemize}
  \item \textbf{Model:} privzeto-zavrni (default-deny) — vse povezave niso dovoljene, razen izrecno dovoljenih.
  \item \textbf{Kaj je dovoljeno:} odjemalci v notranji mreži (INTERNAL) in DMZ lahko dostopajo do interneta; iz interne in DMZ mreže so izrecno dovoljeni dostopi do DMZ storitev (DNS, web, monitoring, AD), WireGuard gostom so dovoljeni upravljalni dostopi; ICMP/ICMPv6 za diagnostične potrebe je omejeno, a omogočeno tam, kjer je potrebno.
  \item \textbf{Kaj je blokirano:} naključni vhodni promet z interneta je privzeto zavrnjen, medsegmentni dostopi so blokirani, razen če obstaja specifično pravilo (npr. upravljanje, AD, monitoring).
  \item \textbf{Hairpin NAT:} notranji in DMZ odjemalci lahko dosežejo javne IPv4 storitve preko hairpin DNAT/SNAT za izbrane services.
  \item \textbf{IPv6:} obstaja vzporedna zbirka pravil za IPv6; za IPv6-only odjemalce so posebej omogočena pravila in NAT66 (NPTv6) za izhod v internet.
\end{itemize}

\subsection{wg-portal (WireGuard portal) in integracija z Active Directory}
\texttt{wg-portal} je spletna aplikacija za upravljanje WireGuard peer konfiguracij, vključno z upravljanjem dovoljenj in samopostrežnim generiranjem konfiguracij za uporabnike. 
V našem okolju \texttt{wg-portal} teče kot upravni vmesnik nad obstoječo PiVPN postavitvijo in je dostopen na \texttt{http://192.168.11.106:8888} (če je nameščen na istem DMZ gostitelju).

\subsubsection{Kaj počne}
\begin{itemize}
  \item Omogoča ustvarjanje in upravljanje peer konfiguracij ter izdajanje QR kod za uporabnike.
  \item Povezuje se na Active Directory za preverjanje uporabnikov in določanje administratorskih pravic preko članstev v AD skupinah.
  \item Upravni vmesnik ne nadomešča same PiVPN postavitve; deluje kot nadgradnja/upravljanje nad obstoječim strežnikom.
\end{itemize}

\subsubsection{LDAP / AD nastavitve (osnutek)}
Za povezavo z AD je običajno potrebno konfigurirati bind uporabnika, base DN in filtre. Primerne nastavitve v \texttt{wgportal.yaml}:
\begin{itemize}
  \item \textbf{LDAP URL:} ldap://192.168.11.201:389 (ali ldaps://... z ustreznimi certifikati)
  \item \textbf{Bind DN:} CN=wgportal\_bind,OU=ServiceAccounts,DC=startup11,DC=local
  \item \textbf{Bind password:} (varno shranjen v konfiguraciji ali vault)
  \item \textbf{User search base:} OU=Users,DC=startup11,DC=local
  \item \textbf{User filter:} \texttt{(|(sAMAccountName=\%s)(userPrincipalName=\%s))} --- poišče uporabnika po sAMAccountName ali UPN
  \item \textbf{Group(s) za admins:} merilo, npr. cn=Admin,OU=Groups,DC=startup11,DC=local
\end{itemize}

\subsubsection{Praktični PowerShell primeri}
Spodaj so osnovni primeri za ustvarjanje bind uporabnika in dodajanje v administratorsko skupino v AD:
\begin{lstlisting}
# Ustvari servisnega uporabnika (prilagodi geslo varno)
New-ADUser -Name "wgportal_bind" -SamAccountName wgportal_bind \
  -AccountPassword (ConvertTo-SecureString 'StrongP@ssw0rd' -AsPlainText -Force) \
  -Enabled $true -Path 'OU=ServiceAccounts,DC=startup11,DC=local'

# Dodeli geslo/izmena obvezna glede na politiko
Set-ADAccountPassword -Identity wgportal_bind -Reset -NewPassword \
  (ConvertTo-SecureString 'StrongP@ssw0rd' -AsPlainText -Force)

# Dodaj bind account v administratorsko skupino za aplikacijo (po potrebi)
Add-ADGroupMember -Identity 'Admin' -Members 'wgportal_bind'
\end{lstlisting}

\subsubsection{Podatkovna baza in pravice}
\texttt{wg-portal} je nameščen na lokaciji \texttt{/opt/wg-portal/}. Privzeta shrmba je SQLite podatkovna baza na \texttt{data/sqlite.db}. 
Moramo dobro poskrbiti da proces ki poganja wg-portal ima adekvatne pravice. V praksi smo morali ustvariti sistemskega uporabnika \texttt{wgportal} 
in nastaviti tega uporabnika kot lastnika vseh datotek v \texttt{/opt/wg-portal}, saj so sicer nastajale napake s pravicami ob zagonu aplikacije. 
Spodaj so primeri ukazov, ki smo jih uporabili (izvedite kot \texttt{root} ali z \texttt{sudo}):
\begin{lstlisting}

# Nastavi lastnistvo in varne pravice za podatkovno mapo
chown -R wgportal:wgportal /opt/wg-portal
chmod -R 750 /opt/wg-portal
\end{lstlisting}

\subsection{Docker Compose primer (snmp\_exporter + Prometheus + Grafana)}
Namestite to na monitoring gostitelja (192.168.11.107) in prilagodite poti/volumne po potrebi.

    image: grafana/grafana:latest
    ports:

    depends_on:
      - prometheus
\end{lstlisting}

\subsection{Konfiguracija Prometheusa (primer scrape job)}
Dodajte `scrape` nalogo, da Prometheus pobira metrike od exporterja (če ne uporabljate Docker Compose DNS, zamenjajte `snmp\_exporter:9116` z `192.168.11.107:9116`):
\begin{lstlisting}
scrape_configs:
  - job_name: 'snmp'
    static_configs:
      - targets: ['192.168.11.1']
    metrics_path: /snmp
    params:
      module: [vyos]
      auth: [vyos_auth]
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target']
        target_label: instance
      - target_label: __address__
        replacement: 192.168.11.107:9116
\end{lstlisting}

  \item Zaženite generator (kot container ali binarko) in dobljeni `snmp.yml` kopirajte na exporter gostitelja v `/etc/snmp\_exporter/snmp.yml`.
\end{itemize}

\subsection{Arhitektura storitve}
\texttt{library-api} je Node.js aplikacija z naslednjimi glavnimi komponentami:
\begin{itemize}

  \item SQLite baza \texttt{library-ha.db},
  \item samopodpisani TLS certifikati v direktoriju \texttt{certs/},
  \item Docker slika in \texttt{docker-compose.yml} za zagon s ponovnim zagonom,
  \item povezava na Active Directory prek LDAP za avtentikacijo in avtorizacijo.
\end{itemize}

Storitev posluša na dveh vratih:
\begin{itemize}
  \item \texttt{3000} za HTTP,
  \item \texttt{3443} za HTTPS in HTTP/2.
\end{itemize}

\subsection{Konfiguracija}
Glavne okoljske spremenljivke v \texttt{server.js} so:

\begin{lstlisting}
DATABASE_FILE=./library-ha.db

SERVER_HTTPS_PORT=3443
SERVER_HOST=0.0.0.0
ETCD_ENDPOINTS=192.168.11.101:2379,192.168.11.102:2379,192.168.11.103:2379

AD_DOMAIN=startup11.local
AD_BASE_DN=DC=startup11,DC=local
AD_LIBRARIAN_GROUP_DN=CN=LIBRARIAN,CN=Users,DC=startup11,DC=local
AD_ADMIN_GROUP_DN=CN=Admin,CN=Users,DC=startup11,DC=local
\end{lstlisting}

\begin{itemize}
  \item \texttt{DATABASE\_FILE} določa pot do lokalne SQLite baze;
  \item \texttt{SERVER\_HTTP\_PORT} in \texttt{SERVER\_HTTPS\_PORT} določata poslušanje HTTP/HTTPS vrat;
  \item \texttt{SERVER\_HOST} mora biti \texttt{0.0.0.0}, da se storitev pravilno veže znotraj Dockerja;
  \item \texttt{ETCD\_ENDPOINTS} poveže aplikacijo z Raft klastrom;
  \item \texttt{AD\_LIBRARIAN\_GROUP\_DN} določa vlogo za pisanje;
  \item \texttt{AD\_ADMIN\_GROUP\_DN} določa administratorsko vlogo iz Active Directory;
  \item ostale spremenljivke določajo LDAP prijavo in povezavo do AD.
\end{itemize}

\section{Raft sinhronizacija}
V tej postavitvi Raft ne izvaja aplikacija sama, temveč \texttt{etcd} na treh vozliščih \texttt{192.168.11.101}, \texttt{192.168.11.102} in \texttt{192.168.11.103}. Aplikacija uporablja \texttt{etcd} za distribuirane ključavnice, osnovni izbor vodje in sinhronizacijo zapisov med lokalnimi SQLite bazami.

Vsaka instanca ima svojo lokalno datoteko \texttt{library-ha.db}. Ko se zapis spremeni, se najprej shrani lokalno, nato pa se v \texttt{etcd} objavi pod ključem \texttt{sync:\textless tip\textgreater:\textless id\textgreater}. Ostale instance te ključe preberejo prek watcherja in posodobijo svoje lokalne podatke.

\subsection{Ključne datoteke}
\begin{itemize}
  \item \texttt{raft-api-integration.js} --- glavna REST implementacija z integracijo v \texttt{etcd}.
  \item \texttt{sync-watcher.js} --- watcher za \texttt{sync:} ključe in aplikacijo sprememb v lokalno bazo.
  \item \texttt{query-ha-db.js} --- hiter pregled vsebine \texttt{library-ha.db}.
  \item \texttt{create-author.js} --- lokalni testni zapis prek REST.
  \item \texttt{manual-replicate.js} --- ročna objava testa replike v \texttt{etcd}.
  \item \texttt{test-graphql.js} --- lokalni testni POST na GraphQL endpoint.
\end{itemize}

\subsection{Kaj mora biti zagnano}
Pred preizkušanjem morajo delovati:
\begin{itemize}
  \item \texttt{etcd} cluster na vseh treh strežnikih,
  \item API storitev na vseh vozliščih,
  \item omrežni dostop do vrat \texttt{2379} in \texttt{2380},
  \item lokalna baza \texttt{library-ha.db} na vsaki instanci.
\end{itemize}

\subsection{Pregled baze}
Za hiter pregled lokalne baze na posameznem vozlišču uporabimo:

\begin{lstlisting}
cd ~/library-api
node query-ha-db.js
\end{lstlisting}

Privzeti SQL ukaz je:

\begin{lstlisting}[language=SQL]
SELECT id, name FROM authors;
\end{lstlisting}

Stanje storitev na vozliščih preverimo z:

\begin{lstlisting}
ssh zanzan01@192.168.11.101 "systemctl status library-api"
ssh zanzan02@192.168.11.102 "systemctl status library-api"
ssh zanzan03@192.168.11.103 "systemctl status library-api"
\end{lstlisting}

\subsection{Preverjanje clustra}
Za pregled članov in stanja clustra uporabimo \texttt{etcdctl}:

\begin{lstlisting}
etcdctl member list --endpoints=http://localhost:2379
etcdctl endpoint status --cluster -w table
\end{lstlisting}

Oddaljen pregled na node1:
\begin{lstlisting}
ssh zanzan01@192.168.11.101 "etcdctl member list --endpoints=http://localhost:2379"
ssh zanzan01@192.168.11.101 "etcdctl endpoint status --cluster -w table"
\end{lstlisting}

\subsection{Testiranje sinhronizacije}
Pomembni testi so naslednji:

\begin{lstlisting}
cd ~/library-api
node create-author.js
node manual-replicate.js
node sync-watcher.js
node test-graphql.js
\end{lstlisting}

Pomen posameznih ukazov:
\begin{itemize}
  \item \texttt{create-author.js} doda testnega avtorja prek REST,
  \item \texttt{manual-replicate.js} ročno objavi \texttt{sync:} zapis v \texttt{etcd},
  \item \texttt{sync-watcher.js} pobere nove \texttt{sync:} ključe in jih uporabi v lokalni bazi,
  \item \texttt{test-graphql.js} preveri, ali je lokalni GraphQL endpoint dosegljiv.
\end{itemize}

Če je sinhronizacija uspela, mora biti rezultat \texttt{node query-ha-db.js} enak na vseh vozliščih.

\subsection{Priporočeni potek}
\begin{enumerate}
  \item Preveri, da je \texttt{etcd} cluster aktiven.
  \item Preveri, da je REST storitev zagnana na vseh treh strežnikih.
  \item Dodaj ali posodobi zapis prek REST ali GraphQL.
  \item Preveri lokalno bazo z \texttt{node query-ha-db.js}.
  \item Po potrebi preveri še \texttt{/health} endpoint in loge storitve.
\end{enumerate}
